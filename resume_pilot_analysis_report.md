> Generated by code scan of the full project workspace.

## 1. Product Overview

**ResumePilot** is an AI-assisted resume analysis and job application tracking platform built for **students, interns, and early-career professionals**. It targets users in Bangladesh who subscribe via the **BDApps** (Robi/Airtel) mobile carrier billing platform.

### Core Workflow (Product Philosophy)

```
Analyze Resume → Improve It → Save a Targeted Version → Attach to Application → Manage Interviews & Follow-ups
```

The app is a focused **workflow tool** — not a job board or a generic career platform. Every feature is designed to serve the above pipeline.

## 6. Core Feature Workflows

### 6.1 Authentication Flow (BDApps OTP)

```
1. User enters phone number (01xxxxxxxxx format)
2. App calls POST /auth/phone/check
   → Backend calls BDApps PHP check_subscription.php
   → If subscribed: return status "subscribed"
   → If not: return "not_subscribed" (user must subscribe via BDApps first)

3. App calls POST /auth/phone/send-otp
   → Backend calls BDApps send_otp.php
   → Returns referenceNo

4. User enters OTP
5. App calls POST /auth/phone/verify-otp
   → Backend calls BDApps verify_otp.php
   → On success: lookup/create User in DB, issue JWT token
   → New users get default UserSettings created automatically

6. JWT stored securely in flutter_secure_storage
7. All subsequent requests use Bearer token
```

**Subscription Guard:** `get_current_active_user` dependency returns HTTP 403 if `user.is_subscribed == false`. Unsubscribed users can only call `/auth/phone/unsubscribe`.

**Token TTL:** 8 days (11,520 minutes), configurable via `ACCESS_TOKEN_EXPIRE_MINUTES`.

### 6.2 Resume Analysis Pipeline

```
User Action: Upload PDF/DOCX or paste text + optional target_role + optional JD text
     │
     ▼
POST /api/v1/resumes/analyze  (HTTP 202 returned immediately)
     │
     ├── 1. [Sync] Extract raw text
     │         - PDF: pdfplumber
     │         - DOCX: python-docx
     │         - Text: direct
     │
     ├── 2. [Sync] Create DB records
     │         - Resume row (raw_text, title, file_type)
     │         - ResumeVersion row ("Original Upload" tag)
     │         - AnalysisResult row (status="processing")
     │
     └── 3. [Background Task] process_resume_background()
               │
               ├── Upload file to Cloudinary (if configured)
               │
               ├── resume_parser.parse_resume_text(raw_text)
               │     → Section detection (contact, summary, experience,
               │       education, skills, projects, certifications)
               │
               ├── resume_scorer.score_resume(parsed_data, raw_text)
               │     ├── check_contact_completeness() → 0–100
               │     ├── check_section_completeness() → 0–100
               │     ├── check_formatting_safety() → 0–100
               │     ├── check_measurable_achievements() → 0–100
               │     ├── check_action_verbs() → 0–100
               │     │
               │     ├── ATS Score = weighted(contact:20, sections:40, formatting:20, metrics:20)
               │     └── Recruiter Score = weighted(metrics:50, verbs:30, sections:20)
               │
               ├── find_missing_keywords(raw_text, target_role, jd_text)
               │
               ├── Generate action_plan from high-severity issues
               │
               └── Update AnalysisResult (status="completed") + Resume.parsed_json

Flutter polls GET /resumes/analyze/{analysis_id}/status until status != "processing"
```

**Analysis Output Fields:**
- `overall_score` (0–100) = avg(ats_score, recruiter_score)
- `ats_score` / `recruiter_score`
- `overall_label` = Poor / Fair / Good / Excellent
- `breakdown` = category score cards
- `issues` = list of {severity, description}
- `missing_keywords` = role-relevant keywords not in resume
- `rewrites` = before/after sentence improvements (MVP: static examples)
- `action_plan` = prioritized steps with point gain estimates

```
Base Resume (uploaded file)
    └── Version 1: "Original Upload" [auto-created on analyze]

  - View version detail and analysis
  - Attach version to a job application
```

### 6.4 Application Tracking Workflow

```
User creates Application → status: "saved"
    │
    ├── Add company, role, source, location, recruiter info
    ├── Attach a Resume Version (optional)
    │
    ├── Status progression:
    │   saved → applied → assessment → hr → technical → final → offer/rejected
    │   Each status change auto-logs a TimelineEvent
    │
    ├── Schedule Interviews
    │   - Round name, type (phone/video/onsite)
    │   - Date, time, timezone, interviewer, meeting link
    │   - Interview status: scheduled → completed/rescheduled/cancelled
    │   - All interview changes logged to Timeline
    │
    ├── Add Reminders (follow-up tasks with scheduled datetime)
    │   - Completing a reminder → Timeline event
    │
    ├── Add Notes (free text, per application)
    │
    └── Timeline → full audit log of all events for this application
```

**Application Detail endpoint** (`GET /applications/{id}`) returns a **mega-payload** composing all related data in one request: application + resume_version info + all interviews + all reminders + all notes + full timeline.

| Concern | Implementation |
|---|---|
| Authentication | BDApps OTP verification (carrier-grade) |
| Token | JWT (HS256), 8-day TTL, stored in `flutter_secure_storage` |
| Subscription gate | `is_subscribed` flag — HTTP 403 on all protected routes if false |
| User isolation | All queries filter by `user_id` from JWT |

| CORS | Configurable via `BACKEND_CORS_ORIGINS` env var |
| Unsubscribe | BDApps-side + marks `is_subscribed=false` (data preserved for re-subscribe) |

## 11. Summary of Core Entities

```
User
 → authenticates via phone (BDApps OTP)
 → uploads Resumes
   → each Resume gets parsed + scored into AnalysisResult
   → user creates ResumeVersions for different roles/companies
 → tracks Applications
   → each Application moves through a status pipeline
   → Interviews, Reminders, Notes are attached
   → Timeline auto-logs all state changes
 → views Dashboard (aggregated overview of all the above)
```

This report covers the complete system as it exists in the codebase today.
